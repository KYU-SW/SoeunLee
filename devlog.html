시행착오 & 해결 전체 기록

01
설계 피벗

React 시도 → Vanilla HTML 복귀

컴포넌트 재사용성을 위해 React로 시작했지만 GitHub Pages 환경과의 충돌로 당일 포기했다.
문제
GitHub Pages는 빌드 결과물(정적 파일)만 서빙 가능. JSX를 브라우저에서 바로 실행하려면 CDN Babel이 필요한데 성능이 나쁘고, Vite/CRA 빌드 도구를 도입하면 "파일 하나 수정 → GitHub에 올리면 바로 배포" 흐름이 깨짐. 포트폴리오 목적에 오버엔지니어링.
Before — React JSX (GitHub Pages에서 동작 안 함)
// Layout.jsx
import React from 'react';
import Navbar from './Navbar';

const Layout = ({ children }) => (
  <div className="app-layout">
    <Navbar />
    <main>{children}</main>
  </div>
);
export default Layout;
After — 단일 index.html, 빌드 없음
<!DOCTYPE html>
<html lang="ko">
<head>
  <style> /* CSS 전부 인라인 */ </style>
</head>
<body>
  <nav class="navbar">...</nav>
  <main class="container">...</main>
  <script> /* JS 전부 인라인 */ </script>
</body>
</html>
배운 점
React는 팀 협업·대규모 상태관리가 필요한 앱에서 진가를 발휘한다. 포트폴리오처럼 단일 페이지 + 간단한 인터랙션은 Vanilla HTML이 더 빠르고 관리하기 쉽다. 도구는 문제에 맞게 선택해야 한다.
02
Feature접근 방식 변경

Firebase 접근 방식 변경 — JS SDK → REST API 직접 호출

Firebase Realtime Database는 그대로 사용하되, 접근 방식을 Firebase JS SDK에서 REST API 직접 호출로 바꿨다. 데이터베이스 자체는 변경 없음.
문제 — Firebase SDK
firebase-app-compat.js + firebase-database-compat.js 두 개의 외부 스크립트를 로드해야 하고, SDK 초기화 코드가 길고 복잡함. 단순 fetch로 할 수 있는 걸 SDK가 추상화해서 오히려 코드량이 늘어남.
Before — Firebase SDK 방식
<script src="https://www.gstatic.com/firebasejs/10.12.2/firebase-app-compat.js"></script>
<script src="https://www.gstatic.com/firebasejs/10.12.2/firebase-database-compat.js"></script>
<script>
  firebase.initializeApp({ apiKey:"...", databaseURL:"...", ... });
  const db = firebase.database();
  db.ref('/comments').push({ name, text, time });
  db.ref('/comments').on('value', snap => { /* 렌더링 */ });
</script>
After — REST API fetch() 직접 호출
const DB = 'https://website-a92c1-default-rtdb.firebaseio.com';

// 댓글 작성
fetch(`${DB}/comments.json`, {
  method: 'POST',
  body: JSON.stringify({ name, text, time: Date.now() })
});

// 댓글 목록
fetch(`${DB}/comments.json`)
  .then(r => r.json())
  .then(data => renderComments(data));

// 방문자 카운터 (일별)
const today = new Date().toISOString().slice(0,10);
fetch(`${DB}/visits/${today}.json`)
  .then(r => r.json())
  .then(n => {
    fetch(`${DB}/visits/${today}.json`,
      { method:'PUT', body: JSON.stringify((n||0)+1) });
  });
결과
외부 스크립트 2개 제거, 초기화 코드 없음, 의존성 0. Firebase REST API는 .json 엔드포인트에 GET/POST/PUT/DELETE만 하면 되므로 fetch 10줄로 동일 기능 구현 가능.

04
AI 실수

Claude가 틀렸던 순간들 — 5가지

AI에게 코드를 요청할 때 잘 못알아 듣는 경우가 많았다.
실수 1 — 엉뚱한 요소를 scatter 처리 "Soeun Lee 글자가 흩어지게 해줘"라고 했더니 AI가 Hero 섹션의 h1이 아닌 nav 상단의 "portfolio" 로고를 scatter 처리했다. 히어로 h1과 nav 로고를 구분하지 못하고 document.querySelector('h1')으로 첫 번째 h1을 잡은 것. "아니 이거 아니야"라고 직접 잡아줘서 id="hero-name"으로 수정.
실수 2 — 사진 회전 방향을 두 번 틀림 프로필 사진이 세로로 나오자 AI가 rotate(90deg)를 적용했으나 반대 방향. 다시 rotate(-90deg)로 수정했으나 이것도 어색함. 결국 CSS 회전을 아예 제거하고 브라우저의 EXIF 자동 처리에 맡기는 것이 정답이었음. 두 번 틀리고 나서야 원인을 제대로 파악.
실수 3 — position:fixed로 사진 배치 프로필 사진 카드를 position:fixed로 배치하고 JS로 뷰포트 좌표를 세팅. 처음엔 제자리처럼 보였지만 스크롤하면 사진이 뷰포트에 고정된 채 콘텐츠 위를 떠다니는 심각한 UX 문제 발생. "카드를 다시 안 누르면 이미지가 같이 내려오잖아" 피드백 후 hero 내부 position:absolute로 교체.
실수 4 — 사진이 글자보다 먼저 등장 scatter 애니메이션 duration이 0.65s인데 사진 등장 setTimeout을 300ms로 설정. 글자가 아직 화면에 있는데 사진이 겹쳐 나타남. "글자가 다 흩어지고 나면 사진이 뜨게"라는 피드백으로 720ms로 수정.
실수 5 — 컨텍스트 윈도우 소진 개발 세션이 길어지면서 AI의 대화 컨텍스트가 가득 차 이전 대화 내용이 압축·소실됨. 새 세션에서는 이전에 논의한 세부 사항을 AI가 기억하지 못해, 인간이 직접 맥락을 다시 설명해야 했음. 한 세션에서 너무 많은 작업을 몰아서 하면 AI의 기억력이 한계에 부딪힌다.
교훈
AI는 코드를 빠르게 생성하지만 실제 UI 맥락(어떤 요소인지, 어떻게 보이는지)을 오해하는 경우가 생긴다. 생성된 코드는 반드시 브라우저에서 직접 확인하고, 틀리면 구체적으로 피드백하는 반복 루프가 핵심. AI가 컨텍스트를 잃기 전에 중요한 결정을 먼저 내리는 것도 중요하다.

FeatureFix ×5

JS 커스텀 스크롤 스냅 — 가장 많이 고친 기능

CSS scroll-snap-type은 키 큰 섹션(About, Projects) 내부에서 스크롤이 완전히 막히는 치명적 단점이 있어서 제외. wheel 이벤트를 직접 핸들링해 구현했지만 수정을 5번 반복했다.
문제 1 — Comments 섹션이 스냅 목록에서 누락
섹션을 querySelectorAll('.content-page')로 수집했는데, Comments 섹션의 클래스명이 comments-section이어서 목록에서 빠짐. Comments로 스냅이 되지 않고 Projects에서 바로 Let's Connect로 건너뜀.
Before
const sections = document.querySelectorAll('.content-page');
// .comments-section은 클래스가 달라서 포함 안 됨
After — ID 배열로 명시
const sections = ['home','about','projects','comments','contact']
  .map(id => document.getElementById(id))
  .filter(Boolean);
문제 2 — 맨 아래까지 스크롤이 안 됨
섹션 위치를 페이지 로드 시점에 1회 캐싱. 댓글이 동적으로 로드되면 섹션들의 실제 위치가 달라지는데, 저장된 값을 그대로 사용해서 타겟이 틀어짐.
Before — 1회 캐싱
// 로드 시 한 번만 계산 → 이후 동적 변화 반영 안 됨
const positions = sections.map(s => s.offsetTop);
After — 매 wheel 이벤트마다 fresh 계산
function pos(s) {
  const r = s.getBoundingClientRect();
  return { top: r.top + window.scrollY, height: s.offsetHeight };
}
// wheel 이벤트 핸들러 안에서 pos(section) 호출
문제 3 — 위로 스크롤 스냅이 안 됨
스냅 시작 시 locked = true로 잠근 뒤, 고정 850ms 후 해제. smooth scroll이 완료되기 전에 해제되어 중간에 끊기거나, 반대로 이미 도착했는데도 잠겨있는 타이밍 충돌이 발생.
Before — 고정 타임아웃
window.scrollTo({ top: targetY, behavior: 'smooth' });
setTimeout(() => { locked = false; }, 850); // 타이밍 불일치
After — scroll 이벤트로 실제 도달 감지
function snapTo(targetY) {
  locked = true;
  window.scrollTo({ top: targetY, behavior: 'smooth' });
  let fallback = setTimeout(() => { locked = false; cleanup(); }, 1500);
  function onScroll() {
    if (Math.abs(window.scrollY - targetY) < 8) {
      locked = false; cleanup();
    }
  }
  function cleanup() {
    clearTimeout(fallback);
    window.removeEventListener('scroll', onScroll);
  }
  window.addEventListener('scroll', onScroll, { passive: true });
}

"Soeun Lee" 클릭 — 글자 scatter + 프로필 사진 등장

h1 "Soeun Lee"와 role 텍스트를 낱글자 span으로 분리 → 클릭 시 랜덤 방향 scatter → 720ms 후 원형 프로필 사진 등장. 재클릭 시 글자 복귀·사진 소멸. 구현 자체보다 디테일을 잡는 데 훨씬 많은 시간이 걸린 기능이다.
문제 1 — span 분리 시 글자가 전부 투명해짐
h1에 -webkit-text-fill-color: transparent로 그라디언트를 적용했는데, 낱글자 span으로 분리하면 자식 span도 transparent를 상속받아 아무것도 보이지 않음.
Before — h1 전체에 그라디언트 (span 분리 불가)
/* CSS */
h1 {
  background: linear-gradient(135deg, #e3a6b9, #f2cbd5);
  -webkit-background-clip: text;
  -webkit-text-fill-color: transparent; /* 자식 span도 상속 → 투명 */
}
After — h1 리셋 후 span마다 그라디언트 직접 적용
// h1의 그라디언트 효과 먼저 해제
h1.style.background = 'none';
h1.style.webkitTextFillColor = 'unset';

// 낱글자 span으로 분리, 각 span에 그라디언트 직접 적용
h1.innerHTML = Array.from(h1.textContent).map(ch =>
  `<span style="display:inline-block;
    background:linear-gradient(135deg,#e3a6b9,#f2cbd5,#f7d7db);
    -webkit-background-clip:text; background-clip:text;
    -webkit-text-fill-color:transparent;
    transition:transform 0.65s cubic-bezier(0.4,0,0.2,1),
               opacity 0.45s ease;"
  >${ch === ' ' ? '&nbsp;' : ch}</span>`
).join('');

/* CSS rotate 없이도 EXIF 방향대로 자동 렌더링 */

FeatureDesign Decision

이스터에그 2종 — 위치·구현 방식 모두 처음과 달라짐

파티클 버스트(Let's Connect)와 TMI 퀴즈(About Me). 기획부터 최종 구현까지 방향이 여러 번 바뀌었다.
파티클 위치 — About Me → Let's Connect 이동
처음엔 파티클 이스터에그를 About Me 섹션에 넣으려 했음. 하지만 차분한 자기소개 섹션에 갑작스러운 파티클 폭발은 맥락 부조화. Let's Connect로 옮기고, About Me엔 퀴즈를 대신 배치.
파티클 소재 — 벚꽃 → 이모지 → Font Awesome 아이콘
벚꽃 이모지(🌸) → 별·하트 이모지(★♥) → Font Awesome fa-star/fa-heart 아이콘 순서로 변경. 이모지는 OS·브라우저마다 렌더링이 달라 일관성이 없음. Font Awesome 아이콘은 CSS color로 색상을 완벽히 제어할 수 있어 최종 선택.
TMI 퀴즈 — 실시간 피드백 → 제출 후 결과 팝업으로 변경
처음엔 라디오 버튼 선택 시 즉시 정답/오답을 표시하는 방식으로 구현. 그런데 답을 바꾸기 어렵고, 퀴즈의 긴장감이 사라짐. "정답입니다/오답입니다가 뜨는 게 아니라 그냥 팝업이 꺼지면서 추가 팝업이 등장하는 거지"라는 피드백으로 변경.
After — 제출 후 일괄 결과 판정
function submitQuiz() {
  // 3문제 모두 선택했는지 먼저 확인
  const allAnswered = quizData.every((_, i) =>
    document.querySelector(`input[name="q${i}"]:checked`)
  );
  if (!allAnswered) { alert('모든 문제를 풀어주세요!'); return; }

  const score = quizData.filter((q, i) => {
    const sel = document.querySelector(`input[name="q${i}"]:checked`);
    return sel && parseInt(sel.value) === q.answer;
  }).length;

  closeQuiz(); // 기존 모달 닫기
  if (score === quizData.length) {
    launchConfetti();  // 전부 정답 → 축포
  } else {
    launchCryRain();   // 하나라도 오답 → 우는 이모지 비
  }
}

주요 설계 결정 & 이유

단일 파일 고수 — React 포기 이유

GitHub Pages는 정적 파일 서빙만 가능. JSX 빌드 없이 React를 CDN Babel로 쓰면 프로덕션 성능이 나쁘고, Vite를 도입하면 배포 복잡도가 급증. 포트폴리오 규모에서는 index.html 단일 파일이 빌드 0, 의존성 0, 수정 후 바로 GitHub에 올리면 배포 완료로 가장 실용적이다.

Firebase 접근 방식 — SDK 대신 REST API 직접 호출

Firebase JS SDK 두 파일을 import하면 수십 KB 번들이 추가되고 초기화 코드가 복잡해짐. Firebase REST API는 {DB_URL}/{path}.json 엔드포인트에 GET/POST/PUT/DELETE 요청만 보내면 되므로, fetch() 10줄로 동일 기능 구현 가능. 외부 스크립트 0개, 초기화 코드 0줄.

CSS scroll-snap 포기 → JS wheel 이벤트 핸들링

CSS scroll-snap-type은 섹션 전환 자체는 쉽지만, About·Projects처럼 키 큰 섹션 안에서 스크롤이 완전히 막히는 치명적 단점이 있다. JS wheel 이벤트를 직접 제어하면 "섹션 높이 > 뷰포트+80일 때는 내부 스크롤 허용, 경계 도달 시에만 다음 섹션으로 스냅"하는 세밀한 로직 구현이 가능. passive:false로 preventDefault()도 허용.

AI 협업 방식 — 검증 루프가 핵심

Claude는 코드를 빠르게 생성하지만 실제 UI 맥락(어떤 요소인지, 실제로 어떻게 보이는지)을 오해하는 경우가 여러 번 있었다. nav 로고 scatter, EXIF 회전, position:fixed 문제 등. 생성된 코드는 반드시 브라우저에서 직접 확인하고 틀리면 구체적으로 피드백하는 반복 루프가 핵심이다. AI가 방향을 잡아주지만 최종 판단은 항상 사람이 한다. 또한 세션이 너무 길어지면 컨텍스트 윈도우가 소진되어 AI가 이전 맥락을 잃어버리므로, 중요한 결정은 세션 초반에 내리는 것이 좋다.
